App Build Prompt

The prompt used to scaffold the demo from the master template.

Paste-ready prompt for Claude Code. Takes the master template and produces a clickable demo of the Wealth Conversation lead magnet on Next.js + Vercel. Scope is a demo, not production.

---

Task

Take the master template and turn it into a working end-to-end demo of the Wealth Conversation lead magnet, deployable on Vercel. Four screens — homepage, questionnaire, data capture, summary — wired with real copy and real segmentation logic, but no backend database, no HubSpot integration, no email sending, no live analytics.

Paths

  • Input: C:\Users\matty\Real Wealth\Real Wealth\master_template\
  • Output: C:\Users\matty\Real Wealth\Real Wealth\wealth_conversation_app\
  • Project reference files: C:\Users\matty\Real Wealth\Real Wealth\ (brand, voice, question catalogue, segmentation, design prompts)

Stack

  • Next.js (App Router) + TypeScript
  • Styling matches whatever the master template uses (Tailwind or CSS modules)
  • Deploy target: Vercel — environment already provisioned
  • No database, no API routes for persistence. Session state lives in React context plus localStorage only.

Read before you start

  1. master_template/README.md — the starting point
  2. Brand Assets/Voice and Tone.md
  3. Question Design Options.md — the 35-question catalogue
  4. Segmentation Design Companion.md and Question Segment Master.xlsx — nine segments, the five-question gate, and the Y/C/— matrix
  5. Design Agent Prompt - Wealth Conversation.md — awareness-check, provocation, and final-page templates
  6. Pitfall Awareness Library.md — the 20 awareness checks and their copy

Scope — demo, not production

Do: wire all four screens end-to-end, load the real 35-question catalogue, run the segment-assignment rules live, skip irrelevant questions per the matrix, show real provocations, land on a segment-tailored summary page.

Don't: build a HubSpot integration (client undecided), fire GA4 events (future), persist to a database, send emails, implement auth. Stub these behind clearly-commented interfaces so the production build can drop in the real implementations later.

What must work

Homepage — hero, tier picker, brand-freedom quote, benefit cards, footer. Tier selection drives question set size. Standard preselected.

Questionnaire — segment-aware. After the five-question gate completes, call the rules engine from the master template, assign a segment, and skip any question marked — for that segment. Progress bar is percentage-based. Three screen types render correctly: question, awareness-check (aware / partial / unaware), provocation card. Provocations fire at their configured positions.

Data capture — one screen: name, email, optional phone, consent tick. Submit writes to session state only. No account, ever.

Summary page — aspiration echo (template blended with keywords from the aspiration question), considered list of 3–6 provocations drawn from what the user encountered, one or two illustrative charts (static SVG is fine — the label and framing matter, not the data), CTA block, FCA footer.

The view-raw-data feature (demo-only)

At the bottom of the summary page, below the FCA footer, add a small link: View the captured data.

Clicking opens a modal (or a route at /demo/raw) that displays the full session as pretty-printed JSON and offers a Download as JSON button. Include:

  • tier (quick / standard / thorough)
  • segmentId and which rule matched
  • Every answer, keyed by question ID
  • Every provocation shown, with timestamp
  • The computed considered list
  • Data capture fields (name, email, phone, consent)
  • Start timestamp, completion timestamp, screen-count

At the top of the modal, a short note: "Demo only. In production this view would not exist — data would flow to the firm's systems via HubSpot or equivalent. This page exists so IT can see the shape of the data we capture."

Segment-specific CTAs on the summary page

The summary-page CTA varies by segment per Segmentation Design Companion.md §7. For the demo, stub each as a placeholder calendar URL with a clear label:

  • S1 Early Accumulator → nurture-list sign-up ("Join our monthly briefing")
  • S2–S5, S7, S8 → standard 20-minute call booking
  • S6 Business Owner, Exit-minded → succession-focused call booking
  • S9 HNW → senior-partner booking
  • Overlay advised_but_looking = true → senior-partner booking regardless of segment

All destination URLs are calendly.com/...placeholder for the demo. Document the mapping in the README.

Non-negotiables

  • No hex values — all colour via tokens from the master template.
  • Voice and Tone banned-word list applies everywhere, including page titles, meta descriptions, error messages, ARIA labels.
  • App never gives advice. Every CTA invites a conversation.
  • WCAG 2.2 AA throughout.
  • Typekit kit ID is a TODO — leave fallbacks and flag clearly.
  • Resume-later: store session in localStorage so a refresh does not lose state. A "Start over" button on the summary page clears it.

Definition of done

  • Deployed to Vercel, public URL in the README.
  • All nine segments can be walked end-to-end — document the five-answer combos that land on each segment so the demo-er can show them deliberately.
  • View the captured data link works, modal opens, JSON displays, download works.
  • Lighthouse accessibility ≥ 95 on all four screens.
  • Typecheck and lint pass.
  • README covers: run locally, deploy to Vercel, the nine segment demo paths, known demo limitations (no HubSpot, no GA4, no persistence, stub calendar URLs), and how a production build would differ.

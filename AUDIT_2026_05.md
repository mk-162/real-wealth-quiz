## 1. Method

The audit treats four classes of risk separately because they fail in different ways:

1. **Numerical correctness.** Tax calculations, allowance tapers, projection arithmetic, gauge math. Failure mode: silently wrong number on the report.
2. **Routing correctness.** Segmentation rules, tile-scoring thresholds, trigger DSL evaluation. Failure mode: wrong copy fires for the wrong user.
3. **Documentation consistency.** Methodology doc, assumptions table, content briefs. Failure mode: doc claims one thing, engine does another.
4. **Edge cases.** All-zero inputs, prefer-not-to-say, very young / very old, partial answers. Failure mode: NaN, undefined, or a confidently wrong number for a missing input.

For each class, the deliverable is a script under `scripts/test-*.ts` (matching the existing `node:test`-via-`tsx` pattern, no new dev dependencies). All scripts run together via `npm run test:audit` and report a single pass/fail summary.

## 5. Findings

### 5.1 Critical — wrong-number-on-report risk

**None found.** Tax math reconciles to the methodology doc's worked examples to within £1 across every section.

### 5.2 Important — wrong-routing-or-misleading-copy risk

**F1 — Two awareness-check / provocation triggers used DSL features the parser doesn't support, so they could never fire on the live site.** _Status: fixed (band-aid)._

| File | Original trigger | Why it never fired | Fix applied |
|---|---|---|---|
| `content/awareness-checks/emergency-fund-sizing.md` | `cash < 3 * essential_monthly_spend` | DSL has no multiplication; `cash` is not an answer key (the slider writes `cash_savings_band`) | Replaced with `cash_savings_band in ['lt25k'] AND essential_monthly_spend in ['3_5to5k', '5to7_5k', '7_5to12k', 'gt12k']` — fires for the same target population (low cash + high essentials), within the limits of the DSL. |
| `content/provocations/spending-ratio.md` | `essential_monthly_spend > 0.5 * income_band_midpoint` | Same — multiplication, computed RHS, neither supported | Replaced with `essential_monthly_spend in ['3_5to5k', '5to7_5k', '7_5to12k', 'gt12k'] AND income_band in ['lt50k', '50to100k']` — captures the high-essentials-low-income population. |

The proper long-term fix is to either extend the trigger DSL to support multiplication and field-vs-field comparisons, or to pre-compute derived fields (e.g. `cash_months_coverage`, `essentials_to_income_ratio`) in the answer shape so the existing DSL can reference them. Tracked under **§5.4 Open**.

**F2 — `scripts/test-tile-scoring-full.ts` was asserting 12 tiles but the simplification dropped to 9.** _Status: fixed._

The pre-S2 simplification (per the comment in `src/lib/compass/types.ts` `TileKey`) dropped state-pension, IHT, and the 12th business/income-mix tile. The full-fixture renderer test was still asserting `tiles.length === 12`, so it had been failing silently for anyone running the whole suite. Updated to `=== 9`. All 99 tests now pass.

**F3 — The S4/S7 segment definitions overlap at age 55–60.** _Status: documented; behaviour preserved._

A 55–60-year-old high earner (`income_band gt200k AND estate ≥ £1m`) technically matches both S4 (Senior Professional / Partner) and S7 (Pre-Retiree Affluent). The rules-engine first-match-wins ordering puts S7 (rank 5) ahead of S4 (rank 6), so this user is routed to "Pre-Retiree Affluent" framing. This appears intentional — at 55+ the user is closer to retirement than to peak senior-professional career — but the segment-description comments in `src/lib/segmentation/types.ts` say S4 is "40–60", which makes the overlap easy to misread. Worth a one-line comment in `types.ts` clarifying the deliberate priority.

### 5.3 Minor — documentation drift / cosmetic

**F4 — Trigger DSL grammar is undocumented from the user's perspective.** Author of a new awareness-check or provocation has no grammar reference outside the inline comment in `src/lib/questionnaire/triggers.ts`. As more triggers are written this becomes a cliff — see open item §5.4-O3.

**F5 — Several legacy fallback paths in `compass/inputs.ts` and `summary/inputs.ts` were defending against the duplicate questions retired earlier in this commit chain.** Cleaned up in commit `149fbff` (preceding this audit).

### 5.4 Open — needs follow-up

**O1 — Trigger DSL needs arithmetic / field-vs-field support, OR derived fields, to express ratio-based triggers correctly.** Recommended path: add pre-computed derived fields (cleanest, additive, doesn't bloat the parser). Examples worth pre-computing on the answer shape:

- `cash_months_coverage = cashSavingsRaw / essentialMonthlySpendRaw`
- `essentials_to_income_ratio = essentialMonthlySpendRaw * 12 / householdGrossIncomeRaw`
- `years_to_retirement = targetRetirementAge - currentAge`
- `total_liquid_wealth = cashSavingsRaw + isaBalanceRaw + giaBalanceRaw`

These would let the existing DSL handle ratio-based triggers via simple `>` / `>=` comparisons. The current band-aided triggers in F1 should be revisited to use the proper signals once derived fields land.

**O2 — IHT and HICBC have no engine functions, only published constants.** The methodology doc mentions both. The constants are pinned in `test-tax-constants.ts`. The awareness-check triggers that consume them (e.g. the `iht-mitigation` trigger using `estate >= 1_000_000`) are exercised in `test-triggers.ts`. But there is no engine-side calculator for "your estate's IHT exposure" or "your HICBC liability this year". If the report is meant to QUOTE an IHT exposure number to clients (not just flag the issue), an engine function and tests for it need to be added.

**O3 — Trigger DSL grammar reference document.** As the trigger corpus grows, content authors need a reference for what's supported. A short file `content/microcopy/trigger-dsl-grammar.md` that lists the supported atom shapes (with examples) would prevent future broken triggers like F1.

**O4 — Reconciliation regression sanity bands are wide.** The BASE_HNW reconciliation in `test-projection-tax.ts` T11 accepts peak total wealth in the £1.5m–£6m range. That window is loose enough to miss a 30% engine-output regression. Tightening to ±10% of the current value (e.g. £5.4m–£6.0m) would catch real model drift, at the cost of needing a deliberate update each time the assumptions change.

### 5.5 Out-of-scope — needs CFP / FCA-authorised reviewer

These cannot be settled by automated tests:

- Voice and tone of every awareness-check expanded copy (29 files)
- Whether the tile-scoring thresholds (e.g. "pension contribution % ≥ age/2 = green") are defensible as planning rules of thumb
- Whether the GIA cost-basis assumption (50% gain fraction) holds up for a typical client's actual portfolio history
- The non-modelled effects listed in methodology §2.11 — fiscal drag, state-pension triple-lock, sequence-of-returns risk, glide path — accept-as-disclosed vs add-to-engine
- Whether banned-phrase regression (`voice:check`) covers the full list a CFP would object to
